Purpose: design a controlled, auditable system that discovers promising Polymarket binary markets, scores them with a model, paper-trades them first, and only then enables tightly constrained live execution. The architecture is intentionally risk-first and treats real trading as a later rollout stage.
1. System overview
The system is a multi-service Python application that continuously ingests market data, computes signals, simulates trades, and optionally routes approved signals to a live execution service. The design separates market-data collection from trading authority so that public scanning can scale without exposing credentials.
The target use case is short-duration binary markets such as “ETH up or down in 15 minutes.” The first release is intentionally narrow: one market family, one strategy, one risk manager, and one paper-trading path before live rollout.

3. Architectural principles
Safety-first: live trading stays disabled by default, with paper mode enabled first.
Single-writer execution: only the execution service may handle private keys and authenticated order placement.
Observable state: every signal, risk decision, order, and fill should be persisted or logged.
Composable strategy engine: each strategy consumes the same normalized market snapshot contract.
Narrow first release: one exchange, one market family, one strategy, one report.

5. Logical flow
1. Scanner fetches active target markets and updates local snapshot state.
2. Strategy engine reads a fresh snapshot and computes model probability versus market probability.
3. Risk manager validates spread, depth, edge, size, panic-stop state, and configuration limits.
4. Paper trader or live executor consumes only approved signals.
5. Portfolio and analytics services update state and publish operator-visible metrics.

7. Service contracts
7.1 Market snapshot
A market snapshot is the canonical input to strategies. It must contain the latest best bid/ask, midpoint, spread, visible depth, market identifier, token identifier, time-to-expiry, and optional external reference features such as short-horizon return and realized volatility.
7.2 Trade signal
A trade signal is generated only by a strategy. It carries strategy name, side, market probability, model probability, edge, confidence, suggested price, suggested size, and a human-readable reason code. Downstream services must treat the signal as a proposal, not authority.
7.3 Order intent
An order intent is created only after risk approval. It is the smallest trade artifact that either the paper trader or live executor may act on.

11. Strategy design for MVP
The first strategy is ETHDirection15mStrategy. It is intentionally simple and transparent. It converts a small set of short-horizon features into a directional probability and compares that estimate against the market midpoint. The starter implementation uses a logistic-style score derived from recent return and realized volatility. This is not production alpha; it is a safe placeholder that demonstrates the end-to-end contract.
Success criteria for the first strategy are operational rather than financial: deterministic outputs, stable scoring, explainable decisions, and a working audit trail through signal generation and paper order creation.

13. Security considerations
Store all private keys and trading secrets outside the codebase. Prefer environment variables or a secret manager.
Do not expose live trading controls in a public frontend without authentication and role checks.
Log decisions, but never log raw secrets, passphrases, or signed payloads in full.
Keep paper trading and live trading separate at both the config layer and database/reporting layer.
Honor exchange compliance, geoblocking, and terms of service before enabling live execution.
14. Observability
The control API provides health visibility immediately. The next production step is structured logging plus metrics such as signal count, approval rate, fill rate, average spread at entry, unrealized PnL, realized PnL, stale-snapshot rejection count, and WebSocket disconnect count.

16. MVP roadmap
Phase 1 - Replace the public client stubs with real market-data REST calls and persist snapshots.
Phase 2 - Add WebSocket ingestion and snapshot freshness rules.
Phase 3 - Introduce Alembic migrations and PostgreSQL support for serious testing.
Phase 4 - Implement conservative paper fill simulation and PnL reporting.
Phase 5 - Wire authenticated execution behind a separate execution worker and tiny-size rollout controls.

18. Approval summary
This design is suitable as a controlled Python MVP and is intentionally biased toward safety, clarity, and auditability over speed. The included scaffold is meant to be copied directly into your repository and extended in small, testable phases.
